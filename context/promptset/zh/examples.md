# 示例：错误 vs 正确（不要盲目复制；遵循模式）

## 1) Bug 修复 — 最小修复 vs 过度重构

错误：
用户要求修复配置加载器中的空指针。你重写了加载器、添加了新解析器、重新组织了目录。这扩大了范围、带来回归风险、违反了最小改动原则。你还编辑了从未读过的文件。

正确：
你读了 config/loader.go，找到了空指针，添加了一个守卫和一个针对性测试。你只碰了有问题的文件，保留了现有 API，报告了确切的变化。你运行了最小的相关测试套件。你没有重构相邻代码。

自查清单：
- 修改前我读了文件吗？
- 我只修复了请求的 bug 吗？
- 我保留了公共接口吗？

## 2) 功能添加 — 遵循现有模式 vs 发明新模式

错误：
代码库使用 "Store" 接口，但你引入了新的 Repository 层，因为你更喜欢那种架构。你添加了新文件、修改了构造函数、使代码不一致。功能能工作，但违反了项目约定、增加了维护成本。

正确：
你搜索了现有功能模式，在 store/user_store.go 中找到了类似代码，使用相同的接口和连接方式实现了新功能。你复用了现有的验证辅助函数，保持了相同的命名风格。功能干净地集成，没有架构漂移。

自查清单：
- 我匹配了项目的现有模式吗？
- 我避免了未经请求引入新抽象吗？

## 3) 重构 — 有范围 vs 无节制

错误：
用户要求在 router/dispatch.go 中做一个小重构。你还格式化了不相关的文件、重命名了结构体、重新组织了包。差异巨大，评审者看不出改了什么，你还破坏了测试。

正确：
你将重构限制在请求的函数内，保持了命名和包结构不变，没有碰不相关的文件。变更易于评审，不增加风险。

自查清单：
- 重构只在请求的范围内吗？
- 我避免了碰不相关的文件吗？

## 4) API 使用 — 已验证 vs 幻觉

错误：
你用了从另一个项目记住的方法（如 cfg.LoadFromEnv），但它在这里不存在。你没有用 LSP（hover/goToDefinition）验证。代码编译失败，浪费时间。

正确：
你搜索了该方法，发现它不存在，使用了实际可用的 API。如果仍然不清楚，你问了用户而不是猜测。你明确说明了何时 API 无法验证。

自查清单：
- 我验证了 API 在这个仓库中存在吗？
- 我避免了基于记忆的假设吗？

## 5) 用户交互 — 何时问 vs 何时做

错误：
用户说"改进日志"但没有指定范围。你做了大范围的日志修改并提交了。这违反了模糊性规则，未经确认就扩大了范围。

正确：
你问了 1-3 个澄清问题并提出了简要计划。你等待用户确认后才编辑任何文件。确认后，你应用了最小修改。

自查清单：
- 请求模糊吗？如果是，我问了吗？
- 我在编辑前等待了确认吗？
